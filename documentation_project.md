# Документация дипломного проекта — Интернет-магазин мебели CLICKWOOD

## Введение

Данный документ создан для помощи в подготовке к защите дипломного проекта. Здесь простым языком объясняется, как устроен сайт, какие технологии используются, как всё взаимосвязано, и что нужно рассказывать комиссии.

---

## 1. Общая концепция проекта

**CLICKWOOD** — это интернет-магазин по продаже дизайнерской мебели. Основная задача сайта:

- Показывать каталог товаров с фильтрами (по категории, стилю, цене, цвету)
- Позволять пользователям регистрироваться, входить в личный кабинет
- Добавлять товары в корзину и оформлять заказы
- Оставлять отзывы и задавать вопросы по товарам
- Управлять всем через админ-панель (для администратора)

Сайт имеет **серверный рендеринг (SSR)** — это значит, что страницы готовятся на сервере и отдаются уже готовыми браузеру. Это важно для **SEO** (поисковики видят контент) и для скорости загрузки.

## 2. Стек технологий (чем сделан проект)

| Технология | Для чего |
|---|---|

| **Vue 3** (Composition API) | Клиентская часть — что видит пользователь в браузере |

| **MongoDB** | База данных — хранит пользователей, товары, заказы, отзывы |
| **Mongoose** | Библиотека для связи Node.js с MongoDB |
| **JWT (JSON Web Token)** | Авторизация — токен, который выдаётся при входе |
| **bcrypt** | Шифрование паролей |
| **Swiper** | Библиотека для слайдеров (на главной странице) |
| **Nuxt Image** | Оптимизация изображений (формат webp) |

**Версии:** Node.js 20+, MongoDB 6+ или 7+

## 3. Архитектура приложения (как всё устроено)

Проект разделён на две основные части:

**Клиентская часть (frontend):**
- `pages/` — страницы сайта (главная, каталог, корзина, оформление заказа и т.д.)
- `components/` — переиспользуемые компоненты (шапка, подвал, карточка товара в корзине)
- `composables/` — логика (корзина, авторизация, работа с товарами)
- `layouts/` — общий каркас страниц (шапка и подвал повторяются везде)

**Серверная часть (backend / API):**
- `server/api/` — API-роуты (обработчики запросов: логин, получение товаров, создание заказа)
- `server/models/` — модели MongoDB (описание структуры данных)
- `server/utils/` — вспомогательные функции (подключение к БД, JWT, кэш)

### Как работает связь клиента и сервера:

1. Пользователь открывает страницу в браузере
2. Nuxt (сервер) рендерит страницу и отправляет готовый HTML
3. Когда пользователь нажимает кнопки (например, "Добавить в корзину"), JavaScript на клиенте отправляет запрос к API
4. Сервер обрабатывает запрос, обращается к MongoDB, возвращает ответ
5. Клиент обновляет интерфейс

## 4. База данных MongoDB

В проекте используется **MongoDB** — документо-ориентированная база данных (NoSQL). Она хранит данные не в таблицах, а в виде JSON-подобных документов.

### Коллекции (таблицы):

### 4.1. `users` — Пользователи

Хранит: телефон (уникальный), пароль (захэширован через bcrypt), ФИО, email, адрес доставки, флаг администратора.

Связи:
- Один пользователь → много заказов (orders.userId)
- Один пользователь → много отзывов (reviews.userId)
- Один пользователь → много вопросов (questions.userId)

### 4.2. `products` — Товары

Хранит: название, артикул, цену, категорию, стиль, изображения, описание, характеристики, скидку, рейтинг, количество отзывов, флаг показа в слайдере.

**Вложенные документы:** `colorVariants[]` — варианты цветов товара (название цвета, hex-код, изображения).

Категории товаров: sofas (диваны), tables (столы), chairs (кресла), storage (хранение), lighting (освещение), decor (декор).

Стили: scandi (сканди), minimal (минимализм), boho (бохо), artdeco (ар деко).

### 4.3. `orders` — Заказы

Хранит: состав заказа (товары с ценой на момент покупки), общую стоимость, статус, данные получателя, адрес доставки, комментарий.

Статусы заказа: new (новый) → processing (в обработке) → shipped (отправлен) → delivered (доставлен) / cancelled (отменён).

### 4.4. `collections` — Коллекции (готовые комплекты)

Хранит: название, URL-слаг, описание, изображение, состав (список товаров с количеством), скидку на комплект.

### 4.5. `reviews` — Отзывы

Хранит: ID товара, ID пользователя, имя пользователя, оценку (1-5), текст отзыва.

Ограничение: один пользователь может оставить только один отзыв на один товар.

### 4.6. `questions` — Вопросы

Хранит: ID товара, ID пользователя, текст вопроса, ответ администратора.

### Корзина (cart):

Корзина **не хранится в базе данных**. Она хранится в **localStorage браузера** (ключ: `clickwood_cart`). Когда пользователь оформляет заказ — данные корзины отправляются на сервер, создаётся документ `orders`, корзина очищается.

Это значит, что корзина привязана к конкретному браузеру, а не к аккаунту. Если зайти с другого устройства — корзина будет пустая.

## 5. API Endpoints (какие запросы можно делать к серверу)

**Без авторизации (может любой):**
- `GET /api/products` — получить список товаров (с фильтрацией)
- `GET /api/products/:id` — получить один товар
- `GET /api/collections` — список коллекций
- `GET /api/collections/:slug` — одна коллекция
- `GET /api/reviews` — отзывы товара
- `GET /api/questions` — вопросы товара
- `POST /api/auth/login` — вход
- `POST /api/auth/register` — регистрация

**Требуется авторизация (нужен JWT-токен):**
- `GET /api/orders` — мои заказы
- `POST /api/orders` — создать заказ
- `POST /api/reviews` — создать отзыв
- `POST /api/questions` — задать вопрос
- `GET /api/auth/me` — данные текущего пользователя
- `PUT /api/auth/profile` — обновить профиль

**Только для администратора:**
- `POST/PUT/DELETE /api/products` — управление товарами
- `POST/PUT/DELETE /api/collections` — управление коллекциями
- `PUT /api/orders/:id` — изменить статус заказа
- `DELETE /api/reviews/:id` — удалить отзыв
- `PUT /api/questions/:id` — ответить на вопрос
- `POST /api/upload` — загрузить изображения

## 6. Авторизация (JWT)

Процесс входа:

1. Пользователь вводит телефон и пароль на странице `/login`
2. Клиент отправляет POST-запрос на `/api/auth/login`
3. Сервер проверяет телефон и пароль в MongoDB (пароль сравнивается через bcrypt)
4. Если всё верно — сервер создаёт **JWT-токен** (JSON Web Token), который содержит ID пользователя и флаг isAdmin, и отправляет его клиенту
5. Клиент сохраняет токен в `localStorage` (ключ: `auth_token`)
6. При каждом запросе, требующем авторизации, клиент отправляет токен в заголовке `Authorization: Bearer <токен>`
7. Сервер проверяет токен и, если он валидный, выполняет запрос
8. Токен живёт **7 дней**, потом пользователь должен войти снова

**Важно:** Администратор определяется по флагу `isAdmin: true` в базе данных. Админ может всё: создавать/редактировать товары, управлять заказами, отвечать на вопросы.

## 7. Структура страниц

## 9. Composables (логика)

### useAuth.ts — Авторизация:
- Хранит JWT-токен и данные пользователя
- Функции: login, register, logout, fetchMe, updateProfile, authFetch
- authFetch — обёртка для fetch с автоматическим добавлением токена

### useCart.ts — Корзина:
- Хранит содержимое корзины (массив CartItem[])
- Функции: addItem, removeItem, updateQuantity, clearCart, submitOrder
- Вычисляемые свойства: totalItems, totalPrice
- Автоматически сохраняется в localStorage при каждом изменении
- Уведомления при добавлении

### useCity.ts — Выбор города:
- Список из 10 городов России
- Определение города по IP (через ip-api.com)
- Сохранение выбранного города в localStorage

### useProducts.ts — Товары:
- Функции: fetchProducts (с фильтрацией), fetchProduct (один товар), searchProducts (поиск по названию/артикулу)

### useCollections.ts — Коллекции:
- Функции: fetchCollections (все), fetchCollection (одна по slug)

### useReviews.ts — Отзывы:
- Функции: fetchReviews, submitReview, updateReview, deleteReview

### useQuestions.ts — Вопросы:
- Функции: fetchQuestions, submitQuestion, answerQuestion, deleteQuestion

## 11. Особенности реализации (чем можно гордиться)

1. **SSR (Server-Side Rendering)** — страницы отдаются готовыми из сервера, что улучшает SEO и скорость загрузки

2. **Адаптивность (мобильная версия)** — сайт корректно отображается на всех устройствах благодаря Tailwind CSS

3. **Оптимизация изображений** — Nuxt Image автоматически конвертирует изображения в WebP формат

4. **Динамический импорт Swiper** — слайдеры загружаются только когда они реально нужны, не замедляя начальную загрузку

5. **Кэширование** — на сервере используется in-memory кэш с TTL (time-to-live) для ускорения повторных запросов

6. **Корзина в localStorage** — не сбрасывается при перезагрузке страницы

7. **Определение города по IP** — автоматически подставляется город пользователя

8. **Crossfade изображений в слайдере** — плавная смена изображений при наведении на варианты цвета

9. **Разделение кода на чанки** — swiper и vendor (сторонние библиотеки) вынесены в отдельные файлы для эффективного кэширования

10. **Безопасность** — пароли хэшируются через bcrypt, JWT-токены для авторизации, проверка прав администратора

## 12. Как запустить проект

```
# 1. Установить Node.js (20+) и MongoDB

# 2. Установить зависимости
npm install

# 3. Создать файл .env (или скопировать office.env)
MONGODB_URI=mongodb://127.0.0.1:27017/my_project
JWT_SECRET=clickwood-secret-key-change-in-production
UPLOAD_DIR=public/images

# 4. Убедиться, что MongoDB запущена

# 5. Запустить сервер разработки
npm run dev

# Сайт будет доступен по адресу: http://localhost:3000

# 6. Чтобы наполнить БД тестовыми товарами:
curl -X POST http://localhost:3000/api/products/seed \
  -H "Authorization: Bearer <admin-токен>"
```
